# Feature Specification: 播放记录手动清理任务

**Feature Branch**: `002-add-play-history-cleanup`
**Created**: 2026-05-11
**Status**: Draft
**Input**: User description: "添加手动任务，通过 Jellyfin IScheduledTask 接口实现四个维护策略：清理2年前过期记录、按用户保留最新N条、全局保留最新N条、数据库优化（VACUUM）。兼容 Jellyfin 10.8.x ~ 10.11.x"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 清理 2 年前过期记录 (Priority: P1)

管理员在 Dashboard Tasks 页面找到"清理过期播放记录"任务，点击 Run 手动执行。任务删除所有用户 2 年前的播放记录，完成后显示清理了"X 条记录"。

**Why this priority**: 这是最常用的清理策略，解决数据库持续膨胀的核心问题——以时间维度删除陈旧数据，逻辑简单、风险最低。

**Independent Test**: 准备包含 1 年前、3 年前记录的数据库，执行任务后验证 3 年前的记录已删除、1 年前的保留。

**Acceptance Scenarios**:

1. **Given** play_history 表中有 1 年前和 3 年前的播放记录，**When** 管理员执行"清理过期播放记录"任务，**Then** 3 年前的记录被删除，1 年前的保留，结果显示"X 条记录已清理"
2. **Given** 所有记录均在 2 年内，**When** 执行任务，**Then** 无记录删除，结果显示"0 条记录已清理"
3. **Given** 数据库文件或表不存在（首次安装），**When** 执行任务，**Then** 任务正常完成，不报错

---

### User Story 2 - 按用户保留最新 10000 条 (Priority: P2)

管理员执行"按用户整理记录"任务，对每个用户各自保留最新的 10000 条播放记录，超出部分删除。活跃用户保留 10000 条，不活跃用户保留其全部少量记录。

**Why this priority**: 解决活跃用户记录过多的问题，同时不误伤记录少的用户。按用户粒度处理更精细。

**Independent Test**: 创建用户 A（有 15000 条）和用户 B（有 500 条），执行任务后验证用户 A 只剩最新 10000 条、用户 B 的 500 条全部保留。

**Acceptance Scenarios**:

1. **Given** 用户 A 有 15000 条播放记录、用户 B 有 500 条，**When** 管理员执行"按用户整理记录"任务，**Then** 用户 A 保留最新 10000 条（最旧的 5000 条被删除），用户 B 的 500 条全部保留
2. **Given** 所有用户的记录数均未超过 10000 条上限，**When** 执行任务，**Then** 无记录删除，结果显示"0 条记录已清理"
3. **Given** 某用户恰好有 10000 条，**When** 执行任务，**Then** 该用户的 10000 条全部保留

---

### User Story 3 - 全局保留最新 10000 条 (Priority: P3)

管理员执行"全局整理记录"任务，整个 play_history 表只保留最新 10000 条播放记录，其余全部删除。适合服务器管理员希望严格控制数据库大小的场景。

**Why this priority**: 激进策略，可能清空冷门用户的记录。作为手动选项提供给有明确需求的管理员，风险由执行者自行承担。任务描述中需明确警告此行为。

**Independent Test**: 在包含多个用户、共 50000 条记录的数据库中执行，验证仅保留最新 10000 条。

**Acceptance Scenarios**:

1. **Given** play_history 表有 50000 条记录（分布在多个用户），**When** 执行"全局整理记录"，**Then** 仅保留最新的 10000 条记录，删除其余 40000 条
2. **Given** 表中记录总数不足 10000 条，**When** 执行任务，**Then** 无记录删除，所有数据保留

---

### User Story 4 - 数据库优化（VACUUM）(Priority: P4)

管理员在执行批量清理后，可以执行"优化数据库"任务，调用 SQLite `VACUUM` 命令重建数据库文件，回收已删除记录所占用的磁盘空间。任务完成后显示数据库优化前后的文件大小变化。

**Why this priority**: SQLite 的 DELETE 不会自动释放磁盘空间——行被标记为已删除，但文件大小不变。VACUUM 是清理大表后释放空间的必要补充操作，但开销较大（需临时额外磁盘空间），适合放在最低优先级由管理员按需手动触发。

**Independent Test**: 数据库中插入 50000 条记录后删除 40000 条，执行 VACUUM 前后记录数据库文件大小，验证文件大小显著减小。

**Acceptance Scenarios**:

1. **Given** 数据库有 50000 条记录后删除了 40000 条（文件仍约 10MB），**When** 管理员执行"优化数据库"任务，**Then** 数据库文件大小显著减小（约 2MB），保留的 10000 条数据完整
2. **Given** 数据库本身无碎片（未经历大量删除），**When** 执行任务，**Then** 任务正常完成，文件大小无明显变化
3. **Given** 数据库文件不存在，**When** 执行任务，**Then** 任务正常完成，不报错

---

### Edge Cases

- 多个任务同时执行时是否互相干扰？（假设：每个任务独立使用数据库事务，WAL 模式下并发安全；VACUUM 需排他锁会自然排队；建议管理员不要同时运行多个任务）
- play_history 表中记录量极大（数十万条）时的性能？（假设：分批删除，每批最多 1000 条，批次间提交事务、报告进度并检查取消令牌）
- 任务执行期间服务器关闭，重启后数据是否一致？（假设：已提交的批次已删除，未提交的批次回滚，不产生部分删除状态）
- "按用户整理"任务中，如果用户数量极多（数百个），遍历所有用户的性能？（假设：先查询所有有记录的用户 ID 列表，再逐用户处理）
- 保留期限为 2 年是否可配置？（本次迭代固定为 2 年，不提供配置项。可在后续迭代中按需添加，遵循 YAGNI 原则）
- VACUUM 期间磁盘空间不足（需临时双倍空间）时如何处理？（假设：SQLite 会在磁盘满时抛出异常，任务显示为失败状态，数据库不受影响）
- VACUUM 与清理任务同时执行时是否冲突？（假设：管理员应避免同时执行多个任务；VACUUM 需要排他锁，会自然排队等待）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供四个独立的任务，均通过 Jellyfin IScheduledTask 接口注册，由 Jellyfin 运行时自动发现
- **FR-002**: 四个任务必须在 Dashboard → Scheduled Tasks 页面中以"Jellyfin Recents"分类显示
- **FR-003**: 四个任务均不设默认触发器（`GetDefaultTriggers()` 返回空），仅支持手动执行
- **FR-004**: 任务 1（"清理过期播放记录"）必须删除 play_history 表中 `played_at` 早于 2 年前的所有记录
- **FR-005**: 任务 2（"按用户整理记录"）必须对每个有播放记录的用户，各自保留最新 10000 条，超出部分删除
- **FR-006**: 任务 3（"全局整理记录"）必须仅保留 play_history 表中最新 10000 条记录，其余全部删除
- **FR-007**: 任务 1-3 必须使用分批删除策略，每批不超过 1000 条记录，批次间提交事务
- **FR-008**: 任务 1-3 必须支持取消操作：每批删除前检查 `CancellationToken`，取消时已提交批次保留
- **FR-009**: 任务 1-3 必须报告进度（0% → 50% 开始删除 → 100% 完成）；任务 4（VACUUM）报告进度（0% → 100%，因 VACUUM 为单次 SQL 命令无法中间分割）
- **FR-010**: 所有任务执行完成后必须记录清理的条目数量，通过 Jellyfin Tasks 框架的结果机制可查看
- **FR-011**: 当数据库文件或 play_history 表不存在时，任务必须优雅跳过而非抛出异常
- **FR-012**: 所有任务必须兼容 Jellyfin 10.8.0 到 10.11.x 版本（targetAbi 10.8.0.0）
- **FR-013**: 任务 3 的描述中必须包含警告信息，提示管理员该操作会影响所有用户
- **FR-014**: 任务 4（"优化数据库"）必须：(a) 执行前通过日志和终端输出当前主数据库文件（`jellyfin-recents.db`）大小；(b) 执行 SQLite `VACUUM` 命令重建数据库文件并自动触发 WAL checkpoint；(c) 执行后再次输出新文件大小及节省的空间差值。任务结果中必须包含"优化前 X MB → 优化后 Y MB，节省 Z MB"。仅关注 `.db` 主文件，不统计 `.db-wal` / `.db-shm` 辅助文件

### Key Entities

- **CleanupTasks（四个任务类）**: 分别实现 IScheduledTask 的任务类，各有不同的 Key、Name、Description 和执行逻辑。任务 1-3 共享相同的分批删除基础设施；任务 4（VACUUM）独立实现，无需分批逻辑
- **TaskResult**: 每个任务执行后的结果，包含已清理的记录数量，由 Jellyfin 任务框架持久化并在 Dashboard 显示

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 任务 1 执行后，play_history 表中不存在 `played_at` 早于 2 年前的记录
- **SC-002**: 任务 2 执行后，每个用户的播放记录数 ≤ 10000（记录数不足的用户保持不变）
- **SC-003**: 任务 3 执行后，play_history 表总记录数 ≤ 10000
- **SC-004**: 单次清理包含 50000 条过期记录的表，总耗时不超过 10 秒（普通硬件条件下）
- **SC-005**: Dashboard Tasks 页面中每个任务的上次执行状态正确反映成功/失败/已取消
- **SC-006**: 清理任务执行期间，正常的播放记录写入操作（INSERT）不受阻塞或错误影响
- **SC-007**: 任务被取消时，已删除的批次已提交、未处理的批次完好保留，不产生数据不一致
- **SC-008**: 所有任务在 Jellyfin 10.8.x 和 10.10.x / 10.11.x 服务器上均可被自动发现并在 Dashboard Tasks 页面显示
- **SC-009**: 任务 4 执行后，在已大量删除记录的数据库中（空闲页 ≥ 30%），主数据库文件大小减少 ≥ 30%

## Assumptions

- 仅清理 play_history 表，不涉及 favorite_record 表（收藏记录量级极小，无需清理）
- 2 年过期阈值固定，暂不提供配置项（遵循 YAGNI，后续按需添加）
- 10000 条上限固定，暂不提供配置项
- 分批删除策略（每批 1000 条）在大多数硬件上不会导致显著的数据库锁定
- SQLite WAL 模式下，删除操作与日常写入的并发互斥开销可忽略
- 任务描述中已包含足够的信息让管理员理解每个任务的行为（特别是任务 3 的全局影响）
- 任务 1-3 共享相同的删除基础设施（分批、进度报告、取消处理），但各自独立实现
- VACUUM 任务独立于清理任务，无分批逻辑，不可取消（VACUUM 执行期间锁定数据库）
- VACUUM 操作期间数据库被独占锁，其他读写操作会短暂阻塞；建议管理员在低峰期执行
